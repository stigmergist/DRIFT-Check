# Control System for Organisational Change
## Explanation & Usage Guide

---

## 1. What this is

This is a **decision control system**, not a delivery framework.

Most approaches focus on *what to do* (e.g. optimise, standardise, transform).  
This model focuses on **when and where those actions are appropriate**.

It helps you avoid:
- Acting too early
- Solving the wrong problem
- Scaling failure
- Wasting effort

---

## 2. The Core Idea

There are two independent dimensions:

### Layer 1 — Context (Should we act, and how?)
- **Proceed** → safe to act
- **Align** → align stakeholders first
- **Probe** → test before acting
- **Stop** → challenge whether to act at all

### Layer 2 — Capability State (What action fits?)
- **Stabilise** → restore control
- **Rationalise** → remove duplication and complexity
- **Optimise** → improve performance

---

## 3. Why this matters

Most failure in organisations is not due to lack of effort or capability.

It is due to:
- Applying the **wrong type of action**
- At the **wrong time**
- In the **wrong context**

Examples:
- Automating unstable systems → faster failure
- Standardising without alignment → fragmentation
- Optimising unclear systems → expensive mistakes

---

## 4. How to use this model (step-by-step)

### Step 1 — Define the capability
Be specific:
- “Customer onboarding process”
- “Legacy platform operations”
- “Store service model”

Avoid vague scopes like “IT” or “Operations”.

---

### Step 2 — Diagnose Context (Layer 1)

Ask: *What is the environment we are operating in?*

Look for observable signals:

- **Proceed**
  - Decisions stick
  - Low friction
  - Consistent execution

- **Align**
  - Conflicting priorities
  - Different interpretations
  - Repeated debates

- **Probe**
  - Unclear cause and effect
  - Disagreement about reality
  - Frequent surprises

- **Stop**
  - Low usage or value
  - Questioned relevance
  - Competing higher priorities

➡️ Select the dominant state (or note if mixed)

---

### Step 3 — Diagnose Capability State (Layer 2)

Ask: *What condition is this capability in?*

- **Stabilise**
  - Frequent incidents
  - Firefighting
  - Unpredictable performance

- **Rationalise**
  - Duplication
  - Inconsistency
  - Coordination overhead

- **Optimise**
  - Stable but slow or costly
  - Underperforming relative to potential

➡️ Select the dominant state

---

### Step 4 — Check alignment

Use the alignment matrix:

- Does the intended action match the actual state?
- If not, what is the risk?

Examples:
- Stabilise + Optimise → very high risk
- Rationalise + Stabilise → limited value
- Optimise + Rationalise → high risk

---

### Step 5 — Apply context override

Context changes how you act:

- **Proceed**
  - Act directly

- **Align**
  - Align before acting

- **Probe**
  - Test before scaling

- **Stop**
  - Challenge the need to act

---

### Step 6 — Formulate decision

Use:

> Context + Capability = Action

Examples:
- Proceed + Stabilise → restore control now
- Align + Rationalise → align before standardising
- Probe + Optimise → test before scaling
- Stop + Optimise → do not invest

---

## 5. Using this across multiple capabilities

Create a simple map:

- X-axis = Capability state
- Y-axis = Context

Plot each capability.

This allows you to see:
- Where you can act now
- Where you need alignment first
- Where to experiment
- Where to stop

---

## 6. Common failure patterns

### Acting too early
Applying change in Align or Probe as if Proceed

### Wrong action type
Optimising instability or rationalising without alignment

### Over-investment
Improving low-value capabilities (Stop context)

### Scaling before understanding
Applying optimisation in Probe contexts

---

## 7. What good looks like

A well-functioning organisation:

- Stabilises before improving
- Aligns before standardising
- Tests before scaling
- Stops low-value work

---

## 8. When to use this model

This model works best when:
- There is confusion about what to do next
- Multiple initiatives are competing
- Change efforts are failing or stalling
- You need to prioritise effort

---

## 9. When this model may struggle

- Highly political environments (requires honest diagnosis)
- Situations with hidden incentives
- When capabilities are poorly defined

---

## 10. Summary

This is a **control system for organisational change**.

It helps answer:

- Should we act?
- What kind of action is appropriate?
- What risks are we taking if we act incorrectly?

And most importantly:

> Are we applying the right action, at the right time, in the right place?
