# Background: Type 1 vs Type 2 Risk, Governance, and Audit Implications

## 1. Core Distinction

Risks behave in two fundamentally different ways:

- **Type 1 (Known / Stable)**  
  Risks where cause and effect are sufficiently understood to allow reliable control.

- **Type 2 (Shifting / Unclear)**  
  Risks where cause and effect are not fully understood and behaviour emerges through use, interaction, or change.

---

## 2. Two Operating Postures

| Posture | Focus | Aim |
|--------|------|-----|
| **Control posture** | Consistency, standardisation | Reduce and manage known risks |
| **Learning posture** | Exploration, observation | Understand and adapt to emerging risks |

**Key tension:**

- Control → makes things consistent  
- Learning → makes differences visible  

---

## 3. Risk Treatment (Same Tools, Different Intent)

| Treatment | Type 1 (Control) | Type 2 (Learning) |
|----------|------------------|-------------------|
| **Reduce** | Primary (controls, prevention) | Premature if used early |
| **Limit** | Secondary (fallbacks) | Primary (contain impact) |
| **Observe** | Confirm controls | Understand behaviour |
| **Accept** | Formal decision | Temporary, while learning |

---

## 4. Key Characteristics

| Aspect | Type 1 | Type 2 |
|-------|--------|--------|
| Cause → effect | Clear | Partial / shifting |
| Behaviour | Predictable | Variable |
| Objective | Control | Understanding |
| Success | Fewer repeat issues | Fewer surprises |
| Failure | Control breakdown | Unexpected outcomes |
| Progress | Consistency increases | Understanding improves |

---

## 5. Mixed Risk Domains

Many risks contain both types:

| Risk | Type 1 | Type 2 |
|-----|--------|--------|
| Data | Governance, access, quality | Usage, recombination, decisions |
| Cyber | Known controls, vulnerabilities | Adversarial behaviour |
| AI | Model controls, pipelines | Emergent behaviour, drift |
| Operations | Known processes | Cross-team interactions |
| Third-party | Contracts, SLAs | Hidden dependencies |

**Pattern:**
- Type 1 = core  
- Type 2 = edge (where most surprises occur)

---

## 6. Systemic Risk

Systemic risk emerges when:

- individual components are **Type 1 (controlled)**  
- but interactions behave like **Type 2 (unpredictable)**  

**Key shift:**
- From managing components  
- To managing **interactions and propagation**

---

## 7. ERM (Enterprise Risk Management)

### Where ERM Works Well
- Type 1 risks  
- Standardisation, comparability, assurance

### Where ERM Struggles
- Type 2 risks  
- Tends to:
  - force early categorisation
  - compress uncertainty into scores
  - create false clarity

### Adapted Use for Type 2

- Use ERM as a **container**, not a precision tool
- Allow:
  - provisional definitions
  - narrative reporting
  - evolving scope
  - explicit uncertainty

---

## 8. Audit Implications

| Aspect | Type 1 Audit | Type 2 Audit |
|-------|--------------|--------------|
| Objective | Assure controls | Expose reality vs assumptions |
| Method | Testing, sampling | Case tracing, scenarios |
| Evidence | Documentation | Real behaviour |
| Output | Assurance opinion | Insight / understanding |

**Risk:**
Using Type 1 audit methods on Type 2 risks creates **false assurance**

---

## 9. Governance (Audit & Risk Committee)

### Type 1
- Structured reporting
- Fixed actions
- Focus on closure

### Type 2
- Flexible discussion
- Focus on surprises
- Track learning, not just completion

**Key question shift:**

- Type 1 → “Are we in control?”  
- Type 2 → “What are we missing?”

---

## 10. Deadlines and Actions

| Type | Deadlines |
|------|----------|
| Type 1 | Fixed, delivery-based |
| Type 2 | Time-boxed learning cycles |

**Failure pattern:**
Type 2 work treated as Type 1 → scope drift, missed deadlines

---

## 11. KRIs (Key Risk Indicators)

- **Type 1 KRIs** → stable, predictive indicators  
- **Type 2 KRIs** → signals of change, anomalies, novelty  

Better distinction than lead/lag:

- **Known signals** → track expected failure  
- **Discovery signals** → reveal new behaviour  

---

## 12. Resilience

| Type | Meaning |
|------|--------|
| Type 1 | Absorb shock and return to normal |
| Type 2 | Adapt and evolve under change |

---

## 13. Capability Framing

Avoid using “maturity” for both.

Better distinction:

- **Type 1** → Reliability (control strength)  
- **Type 2** → Adaptiveness (learning capability)  

**Combined aim:**

> Reliable where understood — Adaptive where not

---

## 14. Role of Specialists

Cyber, data, AI experts:

- Strong in Type 1 (controls, standards)
- Useful in Type 2 only if:
  - exploring behaviour
  - testing under stress

**Failure mode:**
They pull Type 2 risks back into Type 1 framing

---

## 15. Mitigation in Type 2

Mitigation shifts from:

- preventing failure  
→ to  
- containing and learning from it

**Sequence:**

> Contain → Observe → Learn → Then control

---

## 16. Common Failure Patterns

- Treating all risks as Type 1  
- Forcing early certainty  
- Measuring stability instead of change  
- Closing actions before understanding  
- Optimising components while system risk increases  

---

## 17. Core Insight

The challenge is not choosing one model over another.

It is:

> knowing when to apply control  
> and when to allow learning

---

## 18. Practical Test

For any risk, ask:

> Does improvement come from making things more consistent —  
> or from seeing where they behave differently?

- Consistency → Type 1  
- Difference → Type 2
