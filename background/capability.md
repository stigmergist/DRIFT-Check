# Capability, Capability Stacks, and Value Flow

## 1. Capability (Definition)

A capability is:

> the organisation’s ability to repeatedly produce a defined outcome to a consistent standard, through the coherent interaction of people, process, information, technology, and governance in practice.

Constraint:

* If outcomes are inconsistent, the capability is not established.

---

## 2. Capability Components (Enablers)

A capability is enabled by the interaction of:

* People — skills, judgement, behaviour
* Process — how work actually flows
* Information — what is known, shared, and used
* Technology / Tools — what enables or constrains execution
* Governance — how decisions are made and controlled

These do not define the capability individually — only their interaction in producing consistent outcomes does.

---

## 3. Capability Stack (Definition)

A capability stack is:

> a chain of interdependent capabilities, where each capability:

* depends on outputs from capabilities below
* transforms or combines those outputs
* enables capabilities above
* is constrained by the reliability of what it depends on

Core rule:
A capability cannot be more reliable than the capabilities it depends on. If it appears to be more stable, people are compensating are this is masking underlying problems.

---

## 4. Value Flow

Value is created bottom → top:

* Lower layers → create possibility
* Mid layers → integrate and coordinate
* Upper layers → deliver customer-visible outcomes

Customer value is the visible result of the entire stack working.

---

## 5. Core Interaction

Two elements determine behaviour:

* Dependency strength
* Reliability of lower layers

If dependency is high and reliability is variable:

variability propagates upward and becomes amplified

---

## 6. Capability Stack Diagram (Dependencies)

```
[ Customer Outcome Capability ]
            ↑
[ Service / Product Delivery Capability ]
            ↑
[ Coordination / Integration Capability ]
            ↑
[ Execution Capability ]
            ↑
[ Data / Platform / Control Capability ]
```

Each layer:

* consumes from below
* depends on stability below
* enables above

---

## 7. Value Flow + Constraint

```
FOUNDATION (low visibility, high influence)
-----------------------------------------
Data / Platform / Control
    ↓ (enables)
Execution
    ↓
Coordination / Integration
    ↓
Service / Product Delivery
    ↓
Customer Outcome
-----------------------------------------
TOP (high visibility, low control)
```

Value flows upward
Constraints flow upward with it

---

## 8. Failure Propagation

```
Instability at lower layer
        ↓
Inconsistent inputs
        ↓
Workarounds / variation in mid-layer
        ↓
Unpredictable delivery
        ↓
Inconsistent customer experience
```

What appears as a top-layer issue is often lower-layer instability expressed upward.

---

## 9. Key Tension

* Value is seen at the top
* Reliability is created at the bottom

Over-investing at the top while tolerating instability below leads to amplified inconsistency.

---

## 10. Practical Test

For any capability:

* What does it depend on?
* How reliable are those dependencies?
* If they vary, what happens above?

---

## 11. Common Failure Pattern

* Investment focused on top-layer capabilities
* Lower layers remain unstable

Leads to:

optimising instability → improvements do not hold → outcomes drift
