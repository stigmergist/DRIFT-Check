# DRIFT Check

DRIFT Check is a thinking tool for improving **[judgement](judgement.md)** in messy organisational situations.

It keeps analysis anchored in [observable](observation.md) reality. It surfaces **[misfit](misfit.md)** between the [state](state.md) of a system and the actions being applied. It is not a diagnostic engine or answer generator. Its purpose is to help users see clearly and choose better action.

The core operating move is to keep two questions separate but connected. Should we [act now](context.md)? What kind of action fits the **[capability](capability.md)** [state](state.md)? Confusing those questions is a reliable path to misfit.

DRIFT also pushes for minimal, high-value interpretation. The aim is not complete explanation. The aim is to identify the smallest set of interacting elements that explain behaviour well enough to improve the next decision.

## When DRIFT is most useful

DRIFT is especially useful when [signals](signals_and_noise.md) are mixed, coordination is hard, and outcomes are inconsistent. It is even more useful when teams are moving quickly under pressure. In those conditions, people often over-index on activity and under-index on fit.

## What DRIFT does not do

It does not replace domain expertise, governance, or accountability. It does not automate judgement. It helps structure judgement so it is less likely to drift away from observable conditions.

DRIFT is also method-agnostic. It works alongside existing frameworks and governance. See [method_mapping.md](method_mapping.md) for how to use DRIFT with your existing toolkit. For practical workflow guidance, see [docs_companion.md](docs_companion.md).
